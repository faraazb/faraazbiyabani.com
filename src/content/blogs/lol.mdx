## Some context...
I bought a new phone a year ago and since then my old phone has just been lying around. This old phone is a Redmi Note 8 and it is rooted. Having root access on a device unlocks various possibilites.

LinuxDeploy is an app which makes it possible to run a Linux distro in a chroot environment. It also provides both SSH and GUI access to the Linux distro. Inside this Linux distro, it is possible to install a number of self-hostable software. One of them being a DNS sinkhole.

A DNS sinkhole is a software that helps you block a list of domains, typically the ones that serve ads or perform analytics. It does so by acting as the DNS server for our devices, blocking requests to domains on the block list and, resolving and caching the IP address resolutions for other domains by querying an upstream DNS server of our choice such as 1.1.1.1 (Cloudflare) or [another example].

It is then possible to configure a Wifi router or individual devices to use this DNS server; if the router is configured then devices connecting through DHCP will accept the DNS service offered by the router. It is also common for such software to provide options to customize behaviour per client, display query statistics (top blocked domains, percentage of queries blocked), etc. through an admin web interface.

It is worth while to note that a DNS sinkhole is different from a content filtering software which can block/strip webpage content through rules. Also, since DNS works with FQDN, ads served through sub routes of an allowed domain would also not be blocked. Content filtering software can help solve these problems mostly by MITMing the HTTPS traffic. There are two that I know of - Privoxy and Privaxy. I think client side solutions such as a browser extension is simply better for content filtering with respect to both privacy and cost. Also, YouTube ads are a completely different story and cannot be blocked through a DNS sinkhole.

Some popular DNS sinkholes are Pihole, AdGuard Home and Blocky, with Pihole being the most common one. So, that is the first one I decided to setup. I came across the others when I started looking into this.

## Alpine in chroot

The first step was to obtain a Linux environment. I followed instructions from the LinuxDeploy app to create an Alpine instance which is accessible through SSH on my local network. This is my configuration for the LinuxDeploy profile.

LinuxDeploy also needs BusyBox NDK

I chose Alpine for the lightness. I also tried Debian, but I couldn't get SSH to work with it.

Also, a static IP for the device is important as consuming devices will be configured to point at this static IP. This can usually be done through the router settings (look for terms - DHCP, static IP, address reservation) or even on the device itself.

Then I SSHed into the Alpine instance from my computer

## Installing pihole

Alpine is not oficially supported by Pihole!

But, the community has already made scripts, guides and raised discussions regarding installing pihole on baremetal Alpine. TODO Someone has also created a package and so installing it becomes quite simple.

```shell
sudo apk add --repository=http://dl-cdn.alpinelinux.org/alpine/edge/testing pihole
```

and then starting the `pihole` service with
```shell
sudo service pihole start
```

The package install creates a user "pihole", sets up the required directories and files with necessary permissions, sets up logrotate and a cron job to update the pihole domain blocklist database (aka "gravity").
It also creates a init script at `/etc/init.d/pihole` which starts the pihole service after booting.

It still differs from the traditional [`basic-install.sh`](https://google.com) script which also asks you about some basic configuration such as the network interface on which we would like to listen, a password to access the admin interface, TODO, etc. the logging level

[Link](https://example.com)


So I configured some of these by editing `/etc/pihole/pihole.toml`, set the password by running `sudo pihole setpassword` and restarted the pihole service.


Inside `/var/log/pihole`, there's also `FTL.log` and `webserver.log` which can be useful for troubleshooting if something is not working.

## Installing AdGuard Home

Turns out, this is very easy. Especially, if you want to run it from a binary.

I obtained a binary for my architecture arm64/aarch64/armv8 from the GitHub releases, unpacked the tar, enabled execution for the AdGuardHome binary (`chmod +x AdGuardHome`) and run it. It detects that it is the first run and then displays some onboarding configuration through the web UI.

I created a service init file at `/etc/init.d/AdGuardHome` to start the service at startup and I was done "setting it up".

It comes with a default upstream DNS server, and a default block list configured, so it starts working out of the box. It also presents a list of block lists that we can enable.

I configured a password for the admin interface, changed the DNS to Cloudflare, added StevenBlack's list in addition to the default list.

The admin interface also has the ability to classify domains as trackers of various categories which is a nice to have.
